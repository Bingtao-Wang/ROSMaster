## 2. 文档目的

本文档用于持续记录当前 RRT 优化阶段的设计、实验、实现与验证结论。

本阶段目标不是重写 RRT 本体，而是在现有“RRT 检测 frontier -> filter 筛选 -> assigner 分配目标 -> Nav2 执行运动”的链路上，补齐一套可观察、可记录、可回放的探索闭环。

## 3. 当前优化目标

### 3.1 追踪点记录

目标定义：
- 当前真正分配给小车执行的目标点，作为“追踪点”记录。
- RViz 中当前追踪点显示为红球。
- 当小车成功到达该追踪点后，该点状态改为橙色。

说明：
- 追踪点来源必须是 `assigner_ros2.py` 真正发给 Nav2 的点。
- 不能用 `/clicked_point` 代替，因为 `/clicked_point` 只是人工交互输入，不等于真实执行目标。

### 3.2 小车行进轨迹记录

目标定义：
- 记录小车在探索过程中的实际运动轨迹。
- RViz 中显示为一条连续折线，可理解为“红球之间的连线”或“小车真实走过的轨迹线”。

说明：
- 轨迹应来自 `map -> base_link` 的实时位姿，优先使用 TF 获取。
- 不建议每次位姿刷新都直接记点，应增加距离阈值和时间阈值，避免轨迹点过密。

### 3.3 地图保存与路径回放

目标定义：
- 探索阶段保存当前地图。
- 同时保存当前这次探索会话中的追踪点记录和轨迹记录。
- 后续重新载入地图后，让小车重新沿着这次探索得到的“成功追踪点”再走一遍。

已确认决策：
- 回放时，只回放“成功到达过的点”。
- 不回放失败点、不回放仅被分配但未成功到达的点。

### 3.4 探索完成状态

当前结论：
- 原始 RRT 链路里没有正式的“探索完成”状态机。
- 旧 `rrt_save_map.cpp` 只是在“车停够久”时触发保存与退出，这不等于真正探索完成。

本轮实现采用的第一版完成判据：
- 当前没有活跃导航目标
- `/filtered_points` 持续为空
- 上述状态持续超过一段设定时间

当前默认参数：
- `completion_patience = 10.0s`

说明：
- 这个判据先用于“状态发布”和“会话保存判断”，暂不自动强制停机。
- 后续如果实机发现误判，再继续优化为更严格的多条件判据。

### 3.5 探索区域扩展

当前结论：
- 原始实现中，前 5 个点只在启动前生效。
- 一旦 detector 初始化完成，后续新的 `/clicked_point` 会被直接忽略。
- 这就是此前看起来像“有个矩形结界”的根因之一。

本轮修改后：
- 初始仍然使用前 5 个点启动。
- 启动完成后，后续每次再点击 `Publish Point`，都会把该点纳入新的探索包围盒。
- 也就是说，探索进行中可以继续加点，逐步把探索区域往房间更远处扩展。

当前使用建议：
- 第一次先用 4 个角点 + 1 个种子点启动。
- 如果后续发现房间更大、区域不够，只要继续在更远处点击点即可扩展范围。

## 4. 当前系统基础能力盘点

已经具备的基础：
- `assigner_ros2.py` 已经能够给 Nav2 发送真实目标。
- `assigner_ros2.py` 已经发布 `/assigned_goal_marker`，可作为当前目标可视化基础。
- 系统已经可以稳定保存地图到项目 `maps/` 目录。
- 当前位姿可通过 TF 获取，已有 `functions_ros2.py` 中 `robot.getPosition()` 的实现基础。

当前不足：
- 当前目标只显示“此刻目标”，没有历史记录。
- 当前没有“目标成功到达后变色”的状态机。
- 当前没有持续记录机器人真实轨迹。
- 当前没有与地图文件绑定的 trace 会话文件。
- 当前没有专门的 replay 节点去按历史成功点重走路径。

## 5. 总体实现思路

建议把本轮优化拆成两个核心节点能力：

1. `rrt_trace_manager.py`
- 负责记录目标点历史。
- 负责记录机器人轨迹。
- 负责发布 RViz marker。
- 负责保存 trace 会话文件。

2. `trace_replay.py`
- 负责读取保存的 trace 文件。
- 过滤出状态为 `reached` 的追踪点。
- 逐个发送 `NavigateToPose` 目标，让机器人重新走一遍成功路径。

这样拆分的原因：
- 记录逻辑和回放逻辑职责不同，分开后更稳定。
- 不把过多状态机强行塞进 `assigner_ros2.py`，可降低对现有 RRT 主链路的干扰。
- 出现问题时更容易独立排障。

## 6. 推荐的数据记录方式

建议为每次探索创建一个 session，统一使用时间戳命名。

示例：
- 地图：`maps/20260326_223000.yaml` 与 `maps/20260326_223000.pgm`
- 追踪记录：`maps/20260326_223000.trace.json`

建议 trace 文件结构：

```json
{
  "session_id": "20260326_223000",
  "frame_id": "map",
  "map_yaml": "20260326_223000.yaml",
  "goals": [
    {
      "id": 0,
      "x": 1.23,
      "y": 0.45,
      "status": "reached",
      "assigned_at": "2026-03-26 22:30:00 CST",
      "reached_at": "2026-03-26 22:31:12 CST"
    }
  ],
  "trajectory": [
    {
      "x": 0.01,
      "y": 0.00,
      "t": "2026-03-26 22:30:01 CST"
    }
  ]
}
```

说明：
- `goals` 记录任务级目标点历史。
- `trajectory` 记录机器人真实走过的空间轨迹。
- 回放时只读取 `status == reached` 的目标点。

## 7. RViz 可视化方案

建议统一使用 Marker/MarkerArray，可视化规则如下：

- 当前追踪点：大红球
- 已成功到达的追踪点：橙球
- 失败或取消的目标点：灰球
- 小车实际运动轨迹：红色 `LINE_STRIP`
- 当前小车到目标点的指示线：保留，但后续建议统一成红色或橙红色

这样可以在 RViz 中同时看清楚：
- 小车这一刻在追哪个点
- 历史上追过哪些点
- 哪些点已经成功
- 小车实际走过了什么路径

## 8. 分阶段实施计划

### 第 1 阶段：追踪点状态记录与颜色切换

实施内容：
- 在真实分配目标时记录追踪点。
- 维护每个追踪点的状态：`assigned / reached / failed / canceled`。
- RViz 中显示当前红球与历史橙球。

完成标准：
- 小车每拿到一个真实目标，RViz 中能看到新增红球。
- 小车成功到达该目标后，该点自动变为橙球。
- 目标历史能落盘保存。

### 第 2 阶段：轨迹记录与轨迹线显示

实施内容：
- 基于 TF 周期获取小车位姿。
- 以“位移阈值 + 时间阈值”写入轨迹点。
- RViz 显示一条连续轨迹线。

建议初值：
- 最小位移阈值：`0.05 m`
- 最小时间阈值：`0.5 s`

完成标准：
- 小车移动时轨迹线连续增长。
- 静止时不会产生大量重复点。
- 轨迹数据能落盘保存。

### 第 3 阶段：地图保存与 trace 会话文件绑定

实施内容：
- 保存地图时，生成同前缀的 trace 文件。
- 形成一组完整实验资产：`map.pgm + map.yaml + trace.json`。

完成标准：
- 每次实验结束后，能得到一整组同名前缀文件。
- 后续回放只需要输入这一组记录即可。

### 第 4 阶段：成功点回放

实施内容：
- 新建 replay 节点，读取 trace 文件。
- 仅提取 `reached` 状态的追踪点。
- 顺序调用 Nav2 `navigate_to_pose` 重走路径。

完成标准：
- 在加载保存地图并完成重定位后，小车能按成功点顺序重新运行一遍。
- 遇到单点失败时，要有明确策略：重试一次或跳过，后续再定。

## 9. 当前建议的技术选型

### 9.1 关于目标点记录来源

建议：
- 直接挂在 `assigner_ros2.py` 的真实发 goal 位置，或者由新节点订阅一个“目标事件”话题。

原因：
- 这样记录到的才是真正执行目标。
- 与 RRT 的最终决策链路一致。

### 9.2 关于轨迹来源

建议：
- 优先使用 TF 中 `map -> base_link`。

原因：
- 轨迹回放和地图坐标必须统一在 `map` 坐标系下。
- 仅使用 `/odom` 容易累计漂移，不利于后续地图对齐和回放分析。

### 9.3 关于回放对象

已定策略：
- 只回放成功点。

原因：
- 失败点通常意味着该点不可达、定位不稳、局部规划失败或环境临时变化。
- 把失败点也带入回放，会显著降低复现稳定性。

## 14. 第四条实验记录

> 时间：**2026-03-27 00:29:17 CST**  
> 执行者：**Codex**

### 跟踪响应提速优化

问题现象：
- 小车到达一个目标后，前往下一个目标前常有明显停顿。
- 体感上像是在“想很久”才继续走。

本轮定位到的主要瓶颈：
- `filter_ros2.py` 以 `10Hz` 持续对 frontier 做 `MeanShift` 聚类，在 Jetson 上 CPU 开销偏大。
- `filter_ros2.py` 之前会不断堆积非常接近的重复 frontier 点，导致后续聚类和筛选越来越重。
- `assigner_ros2.py` 目标分配周期默认 `0.5s`，在目标刚完成后会带来额外等待。
- `nav2_params.yaml` 中 DWB 本地规划采样量偏大，收到新目标后局部轨迹评估较慢。

本轮已实施优化：
- 在 `scripts/filter_ros2.py` 新增 `cluster_bandwidth`、`min_frontier_separation`、`max_frontier_samples` 参数。
