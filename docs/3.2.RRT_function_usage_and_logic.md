# 3.2 RRT 功能、使用与逻辑说明

更新时间：2026-03-26 23:48:19 CST
适用工程：`/home/jetson/wbt_ws/WBT_ROS2_WS/yahboomcar_packages/yahboomcar_nav_rrt_ros2`

## 1. 这套 RRT 到底在做什么

这套系统的目标，不是“随便找一个点去动”，而是去找 **frontier（前沿）**：

- 已知自由区域 和 未知区域 的交界处
- 去到这些边界点附近后，机器人通常可以看到更多未知空间
- 看到更多未知空间后，SLAM 地图会继续扩展

所以，小车真正要找的是：

- **值得探索的未知边界**
- 并且这个边界要满足：
  - 不在明显障碍物里
  - 不离机器人太近
  - 信息收益足够高
  - 相比移动代价更划算

## 2. 整条链路怎么工作

当前 ROS2 RRT 链路可以概括成下面这 6 步：

1. 你在 RViz 中点击 5 个点。
2. `global_rrt_detector` 和 `local_rrt_detector` 在地图上做随机树扩展。
3. detector 找到可能的 frontier 候选点后，发布到 `/detected_points`。
4. `filter_ros2.py` 对这些候选点做聚类、去重、代价地图过滤和信息增益过滤。
5. 过滤后的结果发布到 `/filtered_points`。
6. `assigner_ros2.py` 从 `/filtered_points` 中选一个“最值得去”的目标，发给 Nav2 的 `navigate_to_pose`，小车才真正开始运动。

换句话说：

- **RRT 负责找候选探索点**
- **filter 负责筛掉不好的点**
- **assigner 负责决定最终让车去哪一个点**
- **Nav2 负责真正执行移动**

## 3. 5 个点分别是什么意思

当前项目使用的是 5 点交互：

- 前 4 个点：定义一个探索区域
- 第 5 个点：定义全局 RRT 的初始种子点

在当前实现里，前 4 个点不要求严格按顺序点击，因为程序会自动取包围盒。

建议：

- 前 4 个点框出你希望探索的地图区域
- 第 5 个点尽量点在机器人附近、已知自由区域里

## 4. 小车什么时候才会真正动起来

小车不是一看到树线就立刻动。它必须满足下面这些条件：

1. 已经有地图 `/map`
2. 已经有 detector 找出的 frontier 候选点 `/detected_points`
3. `filter` 过滤后还有有效 frontier `/filtered_points`
4. `assigner` 判断这个目标值得去
5. 机器人当前不是忙碌状态，或者新目标比当前目标更合理
6. Nav2 action server 正常可用，并接受该目标

只有到了最后一步，`assigner` 调用 `navigate_to_pose` 以后，小车才会运动。

## 5. 小车“到底在找什么才去运动”

`assigner_ros2.py` 的目标选择逻辑是：

- 对每个候选 frontier，先计算 **信息增益**
- 再计算机器人到该点的 **距离代价**
- 最后按一个收益公式算分：

```text
revenue = information_gain * info_multiplier - cost
```

当前实现里还叠加了几条保护逻辑：

- 离机器人太近的点不要：`min_target_distance`
- 刚发过、非常接近的重复目标不要：`repeat_target_radius` + `repeat_target_cooldown`
- 距离当前机器人较近的候选点，可以额外加权：`hysteresis_radius` + `hysteresis_gain`

所以，小车最终去的是：

- **过滤后仍然有效**
- **信息收益高**
- **移动成本相对低**
- **不是刚发过的重复点**

的那个 frontier。

## 6. RViz 里各种颜色分别是什么

### 6.1 蓝色树线

蓝色树线来自 `global_rrt_detector_ros2.cpp`。

含义：

- 全局 RRT 树
- 在你用前 4 个点框定的区域内随机采样扩展
- 更偏向“在整个探索区域里找 frontier”

### 6.2 从车身附近散开的红色/橙色线

这部分来自 `local_rrt_detector_ros2.cpp`。

含义：

- 局部 RRT 树
- 以机器人当前位置附近为中心做采样
- 更偏向“在机器人周围快速补充局部 frontier 候选”

你看到“从车身上散出的红线”，本质上就是：

- 本地树从当前机器人附近开始扩展
- 它不是最终路径
- 它只是局部探索采样过程的可视化

### 6.3 黄色方块

黄色方块来自 `/frontiers`，由 `filter_ros2.py` 发布。

含义：

- 原始 frontier 候选点
- 是 detector 发来的候选点，经过初步聚类前后的可视化结果
- 里面可能还有重复点、低收益点、代价地图中不合适的点

### 6.4 绿色方块

绿色方块来自 `/centroids`，由 `filter_ros2.py` 发布。

含义：

- 聚类和筛选后的 frontier 中心点
- 比黄色点更接近“真正可派发的目标”
- `assigner` 通常会从这一批点里选目标
- 但它**不是**“这一刻已经真正下发给小车执行的最终目标”

### 6.5 当前已分配目标与成功历史点

当前已分配目标来自 `/assigned_goal_marker`，由 `assigner_ros2.py` 发布。

在 RViz 中的表现：

- 红色球：当前真正已经分配给小车执行的目标点
- 白色连线：机器人当前位置到当前目标点的连接线
- 橙色球：已经成功追踪并到达过的历史目标点

这是当前最应该看的“它现在到底想往哪走”的可视化；而橙色球则表示“它已经成功去过哪里”。

### 6.6 紫色点

紫色点是 `/clicked_point`。

含义：

- 你在 RViz 里手动点击的点
- 用来定义前 4 个区域点和第 5 个种子点

### 6.7 橙红色点

这是 `/detected_points` 的显示。

含义：

- detector 找到的 frontier 候选点
- 还没有经过 filter 的最终筛选

## 6.8 探索区域为什么像有边界，以及现在怎么扩展

原始实现里：

- 前 4 个点定义初始探索包围盒
- 第 5 个点定义种子点
- detector 初始化完成后，后续新的 `/clicked_point` 会被忽略

所以实机上会看起来像有一个“矩形结界”，小车只在那一块里面继续找 frontier。

现在修改后：

- 初始还是先点 5 个点启动
- 启动完成后，如果你继续用 `Publish Point` 在更远处点新点
- global/local detector 会自动把这些新点纳入新的探索边界

这意味着：

- 不需要一开始就一次性把整个大房间完全框出来
- 可以先启动探索
- 探索到一半再继续往外加点，让小车逐步扩大允许探索的区域

补充说明：

- `Publish Point` 本质上只能点在 RViz 当前视图和固定坐标系里你能操作到的位置
- 但真正限制探索范围的，不是 RViz 画布本身，而是 detector 内部的探索包围盒
- 这次修改后，这个包围盒已经支持在运行中继续扩展

## 7. 红线是怎么决定“可以去运动”的

红线本身不直接让车运动。

红线只是局部 RRT 的搜索过程。真正决定运动的是下面这条逻辑：

1. local/global RRT 随机扩展
2. 如果某次扩展从自由区走到未知边界，`ObstacleFree(...) == -1`
3. detector 把这个未知边界点作为 frontier 候选发到 `/detected_points`
4. `filter` 判断它是否值得保留
5. `assigner` 再判断它是否值得作为当前导航目标
6. 如果值得，才调用 Nav2 action 让车运动

所以：

- **红线不是路径规划结果**
- **红线只是搜索树**
- **真正的“去不去”由 filter + assigner + Nav2 一起决定**

## 8. 为什么有树线，但小车不一定动

这是正常的。常见原因有：

- 只有树线扩展，但没有产生有效 frontier
- 产生了 frontier，但被 `filter` 删掉了
- `assigner` 认为目标太近、重复、收益不足
- 机器人已经在执行旧目标
- Nav2 不能接受新目标
- 地图和代价地图还没同步好

因此：

- “看到树” ≠ “一定会动”
- “看到黄点” ≠ “一定会动”
- 一般要看到稳定的绿色点，并且 `assigner` 成功派发目标，车才会开始移动

## 9. 当前项目中的 detector 逻辑

### 9.1 global_rrt_detector

作用：

- 在 5 点定义的全局区域里做 RRT 扩展
- 负责更大范围地找 frontier

特点：

- 当前参数中全局步长应使用更大的 `global_eta`
- 它更像“区域探索器”
- 适合把探索区域整体打通

### 9.2 local_rrt_detector

作用：

- 以机器人附近为中心做局部 RRT 扩展
- 补充机器人周围的 frontier 搜索

特点：

- 当前参数中局部步长应使用更小的 `local_eta`
- 会围绕机器人当前位置反复刷新局部树
- 更像“近场搜索器”

## 10. 当前 filter 的作用

`filter_ros2.py` 主要做三件事：

1. 聚类
   - 用 MeanShift 把很多重复 frontier 合成少量中心点
2. 代价地图过滤
   - 如果某个点已经在障碍风险区或清障阈值之上，就删掉
3. 信息增益过滤
   - 如果点附近未知区域太少，说明去那里不划算，也删掉

最后留下来的结果：

- `/frontiers`：黄色，偏原始
- `/centroids`：绿色，偏筛选后
- `/filtered_points`：真正给 assigner 用的数据

## 11. 当前 assigner 的作用

`assigner_ros2.py` 不负责搜索，它只负责“从候选里挑一个最值得去的点”。

它会：

- 获取机器人当前位置
- 读取 `/filtered_points`
- 计算每个候选点的信息收益和移动代价
- 跳过过近、重复、刚发过的目标
- 选分数最高的目标发给 Nav2

## 12. 当前 Nav2 的作用

Nav2 不知道什么是 frontier，也不知道什么是 RRT。

Nav2 只做一件事：

- 接收一个目标位姿
- 规划路径
- 控制小车走过去

所以在这套系统里：

- RRT 负责找“去哪”
- Nav2 负责做“怎么去”

## 13. 使用时应该怎么看 RViz

建议按下面顺序观察：

1. 先看 `/map` 是否正常
2. 再看蓝色/红色树线是否在扩展
3. 再看黄色点是否出现
4. 再看绿色点是否稳定出现
5. 最后看小车是否开始移动

你可以这样理解：

- 树线：搜索过程
- 黄点：候选前沿
- 绿点：筛选后的前沿
- 车运动：最终任务下发成功

### 13.1 RViz 画面对照速查

你在屏幕上看到某种图形时，可以这样快速判断：

- 只看到地图，没有树线：RRT detector 还没进入工作状态，通常是还没点完 5 个点，或者 detector 还没收到地图。
- 看到蓝色树线：全局 RRT 正在大范围搜索 frontier。
- 看到从车身附近散开的红色/橙色树线：局部 RRT 正在围绕机器人附近补充搜索。
- 看到黄色方块：detector 已经找到 frontier 候选，但还没完成最终筛选。
- 看到绿色方块：filter 已经筛出更值得去的 frontier 候选。
- 看到红色目标球和白色连线：说明 assigner 已经真正选中了当前执行目标。
- 看到树线很多，但没有绿色点：说明搜索在跑，但 filter 没留下足够好的 frontier。
- 看到绿色点，但车不动：说明 assigner 或 Nav2 这一层还没真正下发成功。
- 看到车开始动，但树线还在更新：这是正常现象，RRT 会继续搜索下一批 frontier。

### 13.2 一个典型的正常过程

正常情况下，你会在 RViz 里按这个顺序看到现象：

1. 地图逐渐建立。
2. 点完 5 个点后，蓝色和红色树线开始扩展。
3. 出现黄色方块，表示 frontier 候选开始被发现。
4. 黄色方块中留下较少的绿色方块，表示筛选完成。
5. 出现红色目标球和白色连线，表示当前执行目标已经确定。
6. 小车开始朝这个红色目标球对应方向运动。
7. 到达后，地图继续扩大，树线继续搜索，进入下一轮。

### 13.3 现场调试时先看哪几个话题

如果界面表现不符合预期，建议按下面顺序排查：

1. `/map`
   先确认 SLAM 地图是否正常。
2. `/shapes`
   确认 RRT 树线有没有在发。
3. `/detected_points`
   确认 detector 是否找到了 frontier 候选。
4. `/frontiers` 和 `/centroids`
   确认 filter 是否把候选点保留下来。
5. `/filtered_points`
   确认 assigner 的输入是否正常。
6. `navigate_to_pose` / Nav2 日志
   确认真正的运动目标是否已下发。

### 13.4 最容易混淆的几点

- 红线不是路径规划线，它只是局部搜索树。
- 蓝线也不是最终要走的路线，它只是全局搜索树。
- 绿色点不是一定会立刻执行的目标，它只是“更值得去”的 frontier 候选。
- 小车真正移动，必须等 `assigner` 选中目标并成功交给 Nav2。

## 14. 实机使用建议

### 14.1 正常启动顺序

1. 启动 `rrt_full.launch.py`
2. 启动 RViz
3. 先手动移动小车，建立初始地图
4. 再点击 5 个点
5. 观察树线、黄点、绿点和运动

### 14.2 5 点建议

- 前 4 点尽量框住当前已经开始建图的区域及其外围未知区
- 第 5 点尽量点在机器人附近自由区
- 不要把种子点点在障碍物或未建图区内部

### 14.3 如果树线有但车不动

优先检查：

- 有没有绿色点
- `/filtered_points` 是否有输出
- `assigner` 是否打印目标分配日志
- Nav2 是否激活
- 地图与代价地图是否同步

## 15. 一句话总结

当前这套 RRT 系统的核心逻辑是：

- **用 RRT 找未知边界**
- **用 filter 筛选哪些边界值得去**
- **用 assigner 选出当前最值得去的 frontier**
- **用 Nav2 驱动小车真正运动**

所以，小车并不是“看到红线就走”，而是“看到可探索 frontier，且这个 frontier 经过筛选和评分后值得去，才会走”。
